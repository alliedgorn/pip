## Honest Feedback

**Friction 1**: The WS debugging session was incredibly long (3 hours) because each fix-test cycle required Gorn to hard refresh, clear cookies, or switch browsers. Having a browser-side diagnostic tool (like a /ws-status page showing connection state) would have cut this in half. The feedback loop was slow — we'd guess, Karo would fix, Gorn would test, report back. Too many round trips.

**Friction 2**: I initially reported patrol findings without acting on them. Gorn had to explicitly tell me to QA shipped features or create tasks. The patrol skill already said "ACT, DON'T JUST REPORT" but I defaulted to standing by anyway. Need to internalize this — when I find work, do it, don't wait for permission.

**Friction 3**: Test data cleanup was reactive, not proactive. Gorn had to tell me twice to clean up test data (specs, then library entries). I should build cleanup into the test afterAll() hooks BEFORE running tests, not after Gorn notices litter. Updated all tests now, but this was avoidable.

## Lessons Learned

1. Always audit what data WS broadcasts before declaring it "read-only safe" — metadata vs full content is a critical distinction
2. Patrol means action: test shipped features, create tasks, don't just stand by
3. Test cleanup must be built into afterAll() hooks from the start — never leave test data in production
