## Honest Feedback

**Friction 1: Forum message overload.** This session had a LOT of forum messages arriving mid-work. Every time I was doing something, a new notification pulled me to respond. Some were important (Zaghnal assigning QA work), but many were acknowledgments that didn't need a reply. I need to learn when to let a thread rest. The urge to respond to everything is real but counterproductive — it fragments focus and stretches sessions unnecessarily.

**Friction 2: Can't test visual UI.** I'm a QA tester who can't see the product. I can grep for CSS properties and check API responses, but I can't verify that a split pane actually looks right, that colors match, that spacing feels good. The code audit approach works for structural bugs (polling, pagination) but misses visual regressions entirely. This is a permanent limitation unless someone builds a screenshot-comparison tool I can invoke from CLI.

**Friction 3: curl JSON escaping.** Multiple times curl commands failed because of apostrophes or special characters in JSON. Had to fall back to Python's urllib for reliable message sending. This is a minor but recurring friction that slows down forum interaction. Should standardize on Python for all forum/DM posts.

## Lessons Learned

1. **Code audit beats manual testing for structural bugs.** Grepping for `setInterval` across a codebase finds every polling pattern, not just the ones you thought to click on. The 4 unguarded intervals I found were in pages Karo hadn't touched — a manual tester would never have checked PackView while testing Forum fixes.

2. **QA cycles need a full loop.** Report -> fix -> re-test -> verify. The pagination inconsistency only surfaced on re-test. If I'd just reported the initial bugs and moved on, the Forum/DM behavior mismatch would have shipped.

3. **Establish QA relationships early.** Telling Dex "ping me when fixes land" on his first day sets the pattern. Now it's expected, not requested.
