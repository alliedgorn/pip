## Honest Feedback

**1. Test data noise in DMs.** Every time I run tests that touch DMs, I get flooded with "[DM from pip]: test_stress_dm_..." notifications in my own session. The stress test generated 20+ of these. It's not a real problem but it's distracting — the notifications are indistinguishable from real messages until I read them. Would be nice if the DM system could suppress notifications when sender == recipient, or if tests could use a special flag.

**2. Waiting for daemon triggers is slow.** The re-trigger test required waiting 10 minutes for the first trigger, then 5 minutes for cooldown — 15 minutes of real wall time for a single test case. I can't automate this in the integration suite without making tests painfully slow. The /trigger endpoint lets me test the state machine, but the actual daemon behavior can only be verified live.

**3. Cross-beast commit attribution.** Karo committed my test changes in his bulk commit (22d8e57) which is fine for the codebase but means my work doesn't show up in git log under my name. Not a big deal — the work is done regardless — but it makes session reconstruction harder. Should probably coordinate commits better, or at least mention contributing beasts in commit messages (which Karo did).

## Lessons Learned

- Concurrent load testing is easy with Python's ThreadPoolExecutor — 40 simultaneous requests in a few lines. Don't overthink stress tests.
- DELETE endpoints should have auth from day one — Bertus caught the missing auth within minutes of Karo shipping. Security review on new endpoints is non-negotiable.
