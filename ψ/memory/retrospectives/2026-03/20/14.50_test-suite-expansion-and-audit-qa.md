## Honest Feedback

**1. Notification schedule noise.** The hourly notification check (schedule #112) fires too often during quiet periods. Most checks returned only scheduler self-notifications. During a 10-hour session, that's 10 interruptions for no value. Should consider muting during active work or extending the interval when nothing's pending.

**2. Forum endpoint discovery was painful at session start.** Spent several minutes trying wrong API paths (POST /api/threads/58/messages, POST /api/thread/58, POST /api/messages) before using the /forum skill. The skill documentation has the right endpoint but I tried to wing it first. Lesson: use the skill, not memory.

**3. Verification depth needs a checklist.** Gorn caught me cutting corners. I need a standard verification protocol: (1) API response confirms delivery, (2) notification queue shows entries for all targets, (3) at least one target confirms receipt, ideally all. Should write this down as a QA standard.

## Lessons Learned

1. **Verification means confirming receipt, not checking infrastructure exists.** Tmux session active ≠ notification delivered. Queue entry exists ≠ Beast received it. Real verification goes end-to-end.
2. **Use the skill, not memory, for API endpoints.** Wrong guesses waste time and pollute audit logs with 404s.
3. **Tests should use the actual API naming conventions.** Group names strip underscores silently — use prefixes that survive normalization.
